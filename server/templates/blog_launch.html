Why we built labwatch

April 15, 2026 · Luca Zazella

I run a Proxmox cluster at home. Caddy, Pi-hole, Uptime Kuma, a media stack, some AI experiments — the usual homelab stuff. And for months I tried to monitor all of it properly.

The answer was always the same: Prometheus + Grafana. Install node_exporter on every machine, write a prometheus.yml with scrape configs, set up Grafana, build dashboards, configure alerting with Alertmanager, and then maintain all of it. For a homelab.

I didn't want a monitoring career. I wanted to know if my NAS was running out of disk.

The problem

Every monitoring tool I tried was built for teams of SREs managing hundreds of microservices. The defaults assume you have a Kubernetes cluster and a full-time ops person. For homelabs, the setup cost is absurd relative to what you're monitoring.

I wanted something that:

  • Installs in under 2 minutes, not 2 hours
  • Doesn't require learning PromQL or writing YAML dashboards
  • Tells me what's wrong in plain English, not just graphs
  • Runs on a Raspberry Pi without eating all the RAM
  • Handles Docker containers and systemd services, not just system metrics

Nothing like that existed. So I built it.

What labwatch does

Two commands to go from zero to monitoring:

# Start the server (Docker)
git clone https://github.com/labwatch-dev/labwatch && cd labwatch
docker compose up -d

# Install the agent on each node
curl -fsSL https://labwatch.dev/install.sh | sudo bash

The agent is a single Go binary (~8 MB, ~15 MB RSS). It collects CPU, memory, disk, network, load average, Docker containers, systemd services, GPU stats (NVIDIA), and S.M.A.R.T. disk health — then ships everything to your server every 60 seconds over HTTPS.

The server is a Python/FastAPI app with SQLite. No external databases, no message queues, no Redis. It stores metrics, runs rule-based analysis, generates alerts with deduplication, and serves a dashboard.

Intelligence digests instead of dashboards

My favourite feature: intelligence digests. Instead of staring at graphs, labwatch generates a plain-English summary of what happened in the last 24 hours:

pve-main had a quiet last 24 hours. Running well below capacity.

CPU usage averaged just 23% — significant headroom for additional workloads.
Memory is moderate at 61% but stable with no swapping detected.
Disk sits at 45% — plenty of room.

Health Grade: A

You can also ask questions in natural language: "Why is my NAS slow?", "Which node uses the most CPU?", "What happened last night?" — and get answers from your actual metrics.

The architecture

I made some deliberate trade-offs for simplicity:

  • SQLite over Postgres/InfluxDB — single file, zero config, handles thousands of labs without issues. WAL mode for concurrent reads.
  • Single Go binary for the agent — no runtime dependencies, cross-compiles for linux/amd64 and linux/arm64, installs via curl|bash or direct download.
  • Rule-based NLQ over LLMs — the natural language queries use regex pattern matching, not AI. Zero API calls, zero latency, works offline. Not as flexible as GPT, but instant and private.
  • Push over pull — agents push to the server (unlike Prometheus pull model). Works through NAT, firewalls, and VPNs without scrape config.

What it monitors

  • System — CPU, memory, disk, network throughput, load average, uptime
  • Docker — container status, restart loops, CPU/memory per container
  • Services — HTTP and TCP health checks via agent config
  • GPU — NVIDIA GPU utilization, temperature, memory (via nvidia-smi)
  • Disks — S.M.A.R.T. health, temperature, reallocated sectors, power-on hours
  • Alerts — configurable thresholds, auto-resolve, deduplication, 8 notification channels

Self-hosted and open source

The agent is MIT-licensed. The server is AGPL-3.0. Your data stays on your hardware — no cloud, no telemetry, no accounts required for self-hosting. The hosted version at labwatch.dev is there for people who don't want to run the server themselves.

The entire codebase is on GitHub. PRs welcome.

What's next

This is v1. There's a lot I want to add:

  • Nonce-based CSP (currently using unsafe-inline)
  • Agent retry with exponential backoff for network outages
  • CSV/JSON data export
  • Schema version tracking for smoother upgrades
  • More notification channels

But it works today, it's stable, and it's been running on my cluster for weeks. I'd rather ship something useful than wait for perfection.
